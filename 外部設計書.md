# 外部設計書：シンプル・ワンタイム・アラームアプリ

## 1. 概要

本設計書は、「シンプル・ワンタイム・アラームアプリ」が、ユーザーインターフェース（GUI）、データ保存機構、および外部システム（OS/音声）とのやり取りをどのように行うかを定義する。

## 2. システム構成要素

システムは主に以下の3つの主要な構成要素で構成される。

1. **ユーザーインターフェース層 (UI Layer):** ユーザーからの入力受付と設定表示を行う。
2. **データ管理層 (Data Management Layer):** アラーム設定情報（時刻、音）の永続的な保存と読み出しを担当する。
3. **システム制御層 (System Control Layer):** アラームのタイマー管理、時刻チェック、および音声再生などのOSレベルの操作を行う。

## 3. データ設計

### 3.1. 保存データ構造

設定されたアラーム情報は、アプリケーションの再起動後も保持される必要があるため、ファイルに保存する。

*   **保存形式:** JSON形式を採用する。
*   **ファイル名:** `alarm_settings.json`
*   **データ構造:**

```json
{
  "alarm_time": {
    "hour": 0,   // 設定された時 (0-23)
    "minute": 0  // 設定された分 (0-59)
  },
  "alarm_sound": {
    "selection_id": 0 // 選択された音のID（例：0はデフォルト音）
  }
}
```

### 3.2. データフロー（永続化）

| 処理 | 実行タイミング | 処理内容 | 関連要件 |
| :--- | :--- | :--- | :--- |
| **保存** | アラーム設定完了時 | 現在の時刻と選択された音のIDをJSON形式に変換し、`alarm_settings.json`ファイルに書き出す。 | FR-06, NFR-03 |
| **読み込み** | アプリケーション起動時 | `alarm_settings.json`ファイルを読み込み、設定情報をメモリ上に展開する。 | NFR-04 |
| **更新** | 設定変更時 | ユーザーが設定を変更した場合、直ちにメモリ上のデータを更新し、その後、変更された内容をファイルに再保存する。 | FR-01, FR-02 |

## 4. 外部インターフェース設計

### 4.1. ユーザーインターフェース（UI）設計

*   **デザイン原則 (NFR-01):** 最小限の操作と明確なフィードバックを提供する。
*   **入力項目:**
    1. アラーム時刻設定エリア（時、分を選択可能なドロップダウンまたは入力フィールド）。
    2. アラーム音の選択エリア（プリセットされた音のリスト）。
    3. 操作ボタン（設定実行、キャンセル）。
*   **フィードバック:**
    *   設定完了時：「設定が保存されました」というメッセージを表示する。
    *   タイマー停止時：「アラーム設定をキャンセルしました」というメッセージを表示する。
    *   アラーム鳴動時：システム通知または音を再生する。

### 4.2. システム・OS連携インターフェース

システム制御層は、OSレベルの操作を抽象化して実行する。

*   **時刻チェックとタイマー管理 (FR-04, FR-03):**
    *   **設計:** アプリケーションは定期的に（例：1秒ごと）システム時刻と設定時刻を比較するループを維持する。
    *   **動作:** 設定時刻が現在のシステム時刻に一致した時点で、アラーム鳴動処理をトリガーする。
*   **音声再生インターフェース (FR-02, TR-03):**
    *   **設計:** 選択された音ファイル（例：`.mp3`, `.wav`）のパスをシステム制御層に渡し、OSのコマンドラインツール（例：`os.system` または `subprocess`）を通じて外部のメディアプレイヤーまたはOS標準のサウンド機能に再生を依頼する。
    *   **テスト機能 (FR-05):** アラーム鳴動の直前に、同じファイルに対して短い再生コマンドを発行する。

## 5. 処理フロー設計

### 5.1. アプリケーション起動フロー

1. **起動**：アプリケーションが起動する。
2. **データ読み込み**：データ管理層が `alarm_settings.json` を読み込む。
3. **初期化**：読み込んだデータに基づき、UI（設定画面）を初期状態（設定なし）で表示する。
4. **タイマーチェック開始**：システム制御層が、読み込んだ設定時刻に基づき、バックグラウンドで時刻チェックループを開始する。

### 5.2. アラーム設定フロー

1. **入力**：ユーザーが時刻（時、分）と音を選択する。
2. **バリデーション**：入力値が有効な範囲内（例：時 0-23、分 0-59）にあるかチェックする。
3. **データ更新**：有効な入力であれば、メモリ上の設定データを更新する。
4. **永続化**：更新された設定データをJSON形式に変換し、ファイルに上書き保存する。
5. **反映**：UIに「設定完了」のメッセージを表示する。

### 5.3. アラーム鳴動フロー

1. **定期チェック**：システム制御層が継続的にシステム時刻を監視する。
2. **時刻一致**：システム時刻が保存されたアラーム時刻に一致したことを検出する。
3. **鳴動実行**：一致した場合、システム制御層は、データ管理層から対応するアラーム音のファイルパスを取得する。
4. **テスト実行 (FR-05)**：オプションとして、まずテスト音を再生する。
5. **アラーム再生**：テストが完了した場合、指定されたアラーム音を再生する。
6. **状態変更**：アラームが鳴動した後、システムは「アラーム発生済み」の状態に遷移し、次のアラームが発生するまで待機する。

---
**【設計のポイント】**

*   **分離の徹底:** GUI、データ保存、システム操作（OS連携）の各層を明確に分離し、それぞれの責務を限定します。
*   **永続性の担保:** データは必ずファイルに保存されることを保証し、アプリケーションのライフサイクルに関わらず設定が失われないように設計します。
*   **非同期処理の考慮:** アラーム鳴動処理はバックグラウンドで継続的に監視される必要があるため、メインのUIスレッドをブロックしないための非同期処理の設計を念頭に置く必要があります。
